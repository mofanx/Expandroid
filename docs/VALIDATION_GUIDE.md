# 本地化验证工具使用指南

## 概述

为了避免多语言实现过程中遗漏硬编码文本，我们提供了一个自动化验证工具。该工具可以快速检查：
- 资源文件完整性（键的一致性）
- 代码中的硬编码文本
- 资源键统计

## 使用方法

### 运行验证

在项目根目录执行：

```bash
python3 scripts/validate_localization.py
```

### 验证结果

验证工具会输出三个部分的检查结果：

## 验证工具功能

### 1. 资源文件完整性检查

- 检查是否存在至少2个资源文件（默认语言+翻译语言）
- 验证所有语言文件包含相同的资源键
- 报告缺失或额外的资源键

### 2. 硬编码文本检查

- 扫描所有 `.razor` 和 `.cs` 文件
- 查找可能的硬编码英文文本模式
- 自动排除已本地化的调用（`localizationService.GetString`）
- 排除技术术语和代码注释

**检查模式**：
- `"Hello world"` - 包含空格的英文句子
- `"Welcome"` - 长英文单词
- `"Yes!"` - 带标点的短词

**自动排除**：
- 技术术语：`Exception`, `Error`, `string`, `int` 等
- 已本地化调用：包含 `localizationService.GetString` 的行
- 代码注释和技术标识符

### 3. 资源键统计

- 显示每种语言的资源键数量
- 帮助确认资源文件的一致性

## 集成到开发流程

### 建议的使用时机

1. **添加新语言后**：验证新语言资源文件是否完整
2. **修改UI文本后**：确认没有遗漏的硬编码文本
3. **提交代码前**：确保本地化实现完整
4. **CI/CD流程**：可以作为自动化检查的一部分

### GitHub Actions 集成

可以在 `.github/workflows/build.yml` 中添加验证步骤：

```yaml
- name: Validate Localization
  run: python3 scripts/validate_localization.py
```

## 常见问题

### Q: 验证工具报告了误报怎么办？

A: 验证工具会排除技术术语和已本地化的调用。如果仍有误报，可以：
1. 检查是否是技术术语，可以添加到白名单
2. 确认是否真的需要本地化
3. 如果是代码注释或调试信息，可以忽略

### Q: 如何添加新的技术术语到白名单？

A: 编辑 `scripts/validate_localization.py`，在 `technical_terms` 集合中添加：

```python
technical_terms = {
    'Exception', 'Error', 'Warning',
    # 添加你的术语
    'YourTechnicalTerm',
}
```

### Q: 验证工具会检查哪些文件？

A: 工具会检查：
- `src/` 目录下的所有 `.razor` 文件
- `src/` 目录下的所有 `.cs` 文件
- 自动排除 `bin/`, `obj/`, `Designer.cs` 等不需要的文件

### Q: 如何处理平台特定的代码？

A: 平台特定的代码（如 `Platforms/Android/`）可能无法直接使用本地化服务。建议：
1. 使用简短的英文错误消息
2. 在UI层面处理用户友好的本地化消息
3. 保持异常消息的技术性

## 验证结果解读

### ✅ 验证通过

表示：
- 资源文件键完整一致
- 未发现明显的硬编码文本
- 可以安全提交代码

### ❌ 验证失败

表示：
- 资源文件键不一致（缺失或多余）
- 发现可能的硬编码文本
- 需要修复后重新验证

### ⚠️ 警告

表示：
- 发现可能的硬编码文本，但可能是技术术语
- 需要手动确认是否需要本地化

## 最佳实践

1. **定期运行验证**：每次修改UI文本后运行验证
2. **提交前验证**：作为代码提交前的最后检查
3. **团队协作**：确保所有开发人员都知道如何使用验证工具
4. **持续改进**：根据实际情况调整验证规则和白名单

## 技术细节

### 验证算法

1. **资源文件检查**：使用正则表达式提取所有 `data name` 属性
2. **硬编码检查**：使用模式匹配查找可能的英文文本
3. **一致性验证**：比较各语言文件的键集合差异

### 性能

- 验证速度：< 1秒（典型项目）
- 内存占用：< 10MB
- 支持大型项目（1000+ 文件）

## 总结

这个验证工具提供了一种简单高效的方法来确保多语言实现的完整性。通过定期运行验证，可以避免遗漏硬编码文本，确保所有用户可见的文本都正确本地化。
